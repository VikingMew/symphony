# 086 Linear Token From Environment Only

## Goal

Remove `tracker.api_key` from the persisted workflow contract and make Linear API authentication come only from the runtime environment, currently `LINEAR_API_KEY`.

## Context

The current implementation still treats `tracker.api_key` as part of the workflow schema. Even though the UI no longer exposes an API key field, `WorkflowForm` preserves or writes `$LINEAR_API_KEY` into saved workflow versions, `Config.Schema` resolves that field, and diagnostics reports token source based on the active workflow's raw `tracker.api_key`.

That is the wrong ownership model for the current product:

- The Linear token is a runtime secret, not workflow configuration.
- Empty databases should not require an active workflow version just to discover whether `LINEAR_API_KEY` is present.
- Saving workflow settings should not write `api_key` into SQLite.
- Diagnostics and Linear discovery should report the environment token directly.
- `kind` and `endpoint` are fixed Linear runtime facts, not editable workflow fields.

This also explains the confusing operator state where `LINEAR_API_KEY` is set in the process environment, but diagnostics still shows the API token as missing because the active workflow version lacks `tracker.api_key`.

## Scope

- Remove `tracker.api_key` from workflow serialization generated by the settings form.
- Remove `tracker.api_key` from setup-required/default workflow config.
- Change `Config.Schema` so `settings.tracker.api_key` is derived from `System.get_env("LINEAR_API_KEY")`, not from workflow YAML/DB.
- Keep schema parsing tolerant of a stale `tracker.api_key` key only by ignoring it; do not use it, preserve it, or write it back.
- Change `Config.validate!` and Linear client token checks to require `LINEAR_API_KEY` from the process environment.
- Change Linear diagnostics token metadata:
  - source is `env:LINEAR_API_KEY` when present,
  - source is `missing env:LINEAR_API_KEY` when absent,
  - source must not mention workflow literal/fallback/ref.
- Change Linear discovery to work whenever `LINEAR_API_KEY` is present, even if the active workflow is setup-required or has no `tracker.api_key`.
- Split Linear discovery GraphQL into smaller viewer/team/project requests so discovery avoids Linear's query complexity limit.
- Keep `LINEAR_API_KEY` scrubbed from Codex child process environments.
- Update docs to say Linear API key is runtime environment only and is never stored in workflow versions.
- Update tests and helpers so they do not rely on writing `api_key` into workflow packages.

## Non-Goals

- Do not add a web UI field for API keys.
- Do not store encrypted secrets in SQLite in this plan.
- Do not introduce `api_key_secret_ref` as an active runtime feature.
- Do not preserve old workflow token references as a compatibility path.
- Do not make Linear endpoint configurable.
- Do not make tracker kind configurable.
- Do not change how project slug, state names, or assignee are configured.

## Design Decisions To Validate During Implementation

- `LINEAR_API_KEY` should be the only supported token source for this alpha-stage implementation.
- Stale persisted workflow rows that contain `tracker.api_key` should not crash parsing, but the value should be ignored and omitted on the next save.
- Diagnostics should be able to show token presence while the app is still in setup-required mode.
- Linear discovery should not need an active workflow, because its purpose is to help create the first correct settings.
- Token fingerprint and length can remain in diagnostics, but they must be computed from the environment value only.

## Acceptance Criteria

- [x] Saving `/settings/workflow` never writes `tracker.api_key` into the generated workflow package.
- [x] Setup-required workflow config does not contain `tracker.api_key`.
- [x] `Config.settings!().tracker.api_key` equals `System.get_env("LINEAR_API_KEY")` when the env var is present.
- [x] `Config.settings!().tracker.api_key` is `nil` when `LINEAR_API_KEY` is missing.
- [x] A stale workflow package with `tracker.api_key: "token"` does not make runtime use `"token"`.
- [x] Linear diagnostics shows token source as `env:LINEAR_API_KEY` when the env var is present.
- [x] Linear diagnostics shows token source as `missing env:LINEAR_API_KEY` when the env var is absent.
- [x] Linear diagnostics and discovery do not require an active workflow `tracker.api_key`.
- [x] Linear client requests use the environment token.
- [x] Codex child process tests still prove `LINEAR_API_KEY` is not inherited by Codex.
- [x] Docs no longer instruct users to put `api_key` in workflow YAML.
- [x] Tests no longer assert that workflow saves include `api_key`.

## Implementation Notes

- Start at `Config.Schema.normalize_settings/1`, where `tracker.api_key` is currently resolved from the workflow value with `LINEAR_API_KEY` fallback.
- Remove `WorkflowForm.tracker_api_key/1` and the `put_optional_path(["tracker", "api_key"], ...)` write.
- Update `Workflow.setup_required_workflow/1` to omit `api_key`.
- Update `Linear.Diagnostics.raw_api_key_metadata/0` to read environment metadata directly.
- Update `Linear.Discovery.discovery_tracker/0` so setup-required mode can still provide fixed kind/endpoint plus env token.
- Do not add a data migration for old rows; alpha-stage behavior should ignore stale `api_key` keys and overwrite them on the next save.
